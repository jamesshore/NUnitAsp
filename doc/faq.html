Frequently Asked Questions

The NUnitAsp team sees frequently asked questions as indicators of problems with NUnitAsp. We strive to eliminate FAQs by improving NUnitAsp itself, so this isn't the best place to start for general information about NUnitAsp. For an overview, see the NUnitAsp QuickStart Guide. For a detailed tutorial, see the NUnitAsp Tutorial. For installation issues, see the Installation Guide. NUnitAsp is available for download at the NUnitAsp web site.

Q: How do I test if a page has redirected properly?

A: Use the Browser.CurrentUrl property to look at the current URL or the CurrentWebForm.AspId property to look at the ASP.NET ID of the current web form.

Q: How do I test user controls / data grids?

A: See the section titled "User Controls, Data Grids, and Other Control Nesting" in the Quick-Start Guide.

Q: When will NUnitAsp support frames / Javascript / pop-up windows / more HTML controls?

A: NUnitAsp is a unit testing tool. Its purpose is to test logic in ASP.NET code-behind classes. When you write an NUnitAsp test, focus on testing the behavior of a specific page or even a specific user control. We expect you to ignore frames, Javascript, and other client-side issues and focus on ASP.NET's interaction with the server.

The NUnitAsp team agrees that client-side support would make it easier to test the whole UI. Given our limited manpower, we have decided to focus on the code-behind unit-testing aspect of NUnitAsp instead. If these features are important to you, well-written and tested code contributions are always welcome. You may also fund development.

Q: When I try to compile NUnitAsp in VS.NET, I get a "cryptographic failure" error. What's wrong?

A: The NUnitAsp DLL is strong-named, which means it must be signed by a cryptographic key. If you try to compile without a key, you'll get this error message:

Cryptographic failure while signing assembly 'C:\projects\NUnitAsp\source\NUnitAsp\obj\Debug\NUnitAsp.dll' -- 'The key container name 'NUnitAsp' does not exist'

We don't include a key in the distribution, but the NUnitAsp build script automatically generates the correct kind of key for you. Open a Visual Studio .NET command prompt and type 'nant' in the root NUnitAsp directory to run it. You only need to do this the first time you compile.

Q: When I try to compile NUnitAsp with NAnt, the build says "External program returned errors". What's wrong?

A: In order to sign the NUnitAsp DLL, the build uses a program called "sn.exe". If it's not available, your build will fail with this error:

keypair:
[exec] sn -k NUnitAsp.snk

BUILD FAILED
E:\devtools\nunitasp\NUnitAsp.build(50,4): External program returned errors, see build log for details.
Try 'nant -help' for more information

To correct this, make sure sn.exe is in your path or in the DotNetToolDir environment variable path. If you don't have a DotNetToolDir environment variable, the build script will set it to a default, but this may be wrong. Make sure it's set properly.

Q: When will NUnitAsp stop requiring XHTML?

A: Done! As of version 1.4, NUnitAsp no longer requires XHTML.

Q: When will NUnitAsp support NUnit v2.1?

A: Done! As of version 1.4.1, NUnitAsp supports NUnit v2.1.4.

Q: When will NUnitAsp support Visual Studio .NET 2003?

A: Done! As of version 1.4, NUnitAsp works with Visual Studio .NET 2003. The NUnitAsp solution will remain on the original version of Visual Studio .NET unless someone sends Jim Shore a complementary copy of VS.NET 2003. (Microsoft: hint, hint.)
